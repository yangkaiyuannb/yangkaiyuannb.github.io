Git的介绍与应用

发表于 2021-07-14 分类于 版本控制器-Git

Git简介

Git 是一个免费的，开源的分布式的版本控制工具，与其区别的是集中式版本控制工具。可以快速高效的处理从小型到大型的各种项目。

Git易于学习，占地面积小，性能极快，具有廉价的本地库（本地库在本地磁盘上），方便的暂存区域和多个工作流的分支特性。他的性能优于Subversion, CVS, Perforce 和 ClearCase 等版本控制工具。

1.1 什么是版本控制？

版本控制是一种记录文件内容变化，以便将来查询特定版本修订情况的系统。

版本控制其实最重要的是可以记录文件的历史修改记录，能让用户查看历史版本，方便版本切换。

1.2 版本控制工具

CSV, SVN(Subversion)， VSS …

集中化的版本控制系统如上述，都有一个单一的集中管理服务器， 保存所有文件的修订版， 而协同工作的人们都通过客户端连到这台服务器， 取出最新的文件或者提交更新， 这是版本控制系统的标准做法。

这种做法好处是每个人都可以在一定程度上看到项目上的其他人正在做什么， 而管理员也可以很轻松的掌握开发者的权限， 并且管理一个集中化的版本控制系统， 要远比在各个客户端上维护本地数据库来的容易轻松。

这种做法的坏处是中央服务器单点故障， 如果中央服务器宕机一小时，那么在这一小时中，谁都无法提交更新， 也就无法协同工作。虽然在本地可以写代码，但是无法将代码提交到中央服务器也就意味着新代码无法获得版本更新的历史记录。

Git， Mercurial， Bazaar, Darcs……

分布式版本控制工具如上述，像Git这种分布式的版本控制工具，不存在所谓的中央服务器，或者说所有客户端都是一个服务器，客户端提取的不是最新版的文件快照，而是把代码仓库完整的镜像下来（本地库）。这样任何一个协同工作用的文件发生故障，事后都可以用其他客户端的本地仓库进行恢复。因为每个客户端的每一次文件提取操作，实际上都是对整个文件仓库的完整备份。用户写完代码后可以直接推送到远程库（写代码时注意，永远保证远程库的代码是最新的），然后其他用户可以从远程库进行克隆代码。如果远程库挂掉，用户也可以在本地进行版本控制获得版本更新历史记录。只是不能把代码推送到远程库。

分布式版本控制系统出现后，解决了集中式版本控制系统的缺陷。

  • 服务器断网的情况下也可以进行开发（版本控制是本地进行的）。

  • 每个客户端保存的也都是整个的完整的项目（包含历史记录，更加安全）。

1.3 Git工作机制

1.31 工作机制

工作区是你写代码的地方（代码存放磁盘目录的位置）

暂存区让git知道你写了代码需要把代码从工作区添加到暂存区（git add），暂存区里代码是临时存储，在暂存区的代码可以删除，在这个区域没有生成代码的历史版本。要想获得历史版本需要把代码从暂存区提交到本地库（git commit）

本地库的代码是无法删除的。代码到本地库会有相应的历史版本。在本地库写完之后还可以将代码推送到代码托管中心-远程库（git push）

3.1 设置用户签名

签名的作用是区分不同操作者的身份，用户的签名信息在每一个版本的提交信息中都能够查看到，由此确定本次提交是谁做的。首次安装必须提交签名否则会无法提交代码。

注意 ： 这里设置用户签名和将来登录GitHub（或者其他代码托管中心） 的账号没有任何关系。

Git分支

什么是git的分支

在版本控制过程中，同时推进多个任务，我们可以创建每个任务的单独分支。 使用分支意味着程序员可以把自己的工作从开发主线上分离开来，开发自己分支的时候不会影响主线分支的运行。 可以理解为分支即是副本，一个分支就是一个单独的副本（分支的底层是指针的引用）。

分支特性

可以同时并行推进多个功能的开发，提高开发效率。

各个分支在开发过程中，如果某个分支开发失败，不会对其他分支有任何影响。失败了删除分支重新开始即可。
